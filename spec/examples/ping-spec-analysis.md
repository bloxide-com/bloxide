# Example: Ping Spec → Code Mapping

This document shows how each section of a blox spec maps to concrete code artifacts. Use this as a reference when writing specs for your own bloxes.

---

## Spec Section → Code Artifact Table

| Spec Section | Code Artifact | File |
|--------------|---------------|------|
| State Hierarchy diagram | `PingState` enum | `crates/bloxes/ping/src/spec.rs` |
| States table | `is_leaf()`, `parent()` impls | Generated by `#[derive(StateTopology)]` |
| Events table | `transitions!` rules | `crates/bloxes/ping/src/spec.rs` |
| Context struct | `PingCtx` definition | `crates/bloxes/ping/src/ctx.rs` |
| Context annotations | Generated accessor impls | Generated by `#[derive(BloxCtx)]` |
| Message Contracts (receives) | Event enum + transitions | `events.rs`, `spec.rs` |
| Message Contracts (sends) | Action functions | `crates/actions/ping-pong-actions/src/lib.rs` |
| Entry/Exit Actions | `on_entry`, `on_exit` slices | `crates/bloxes/ping/src/spec.rs` |
| Constants | `const` items | `crates/bloxes/ping/src/lib.rs` |
| Acceptance Criteria | Test functions | `crates/bloxes/ping/src/tests.rs` |

## Walkthrough: One Transition Rule

### Spec (from ping.md)

```markdown
| Event | Handled by | Rule pattern | Guard outcome | Side effects |
|-------|-----------|--------------|--------------|--------------|
| `PingPongMsg::Pong(_)` | `Active` | Action-Then-Guard | `Error` if failed, `Done` if done, `Paused` if pause, else `Active` | log, forward_ping |
```

### Code (from spec.rs)

```rust
const ACTIVE_FNS: StateFns<Self> = StateFns {
    on_entry: &[increment_round, Self::log_round, send_initial_ping],
    on_exit: &[],
    transitions: transitions![
        PingPongMsg::Pong(_) => {
            actions [Self::log_pong_received, Self::forward_ping]
            guard(ctx, results) {
                results.any_failed()                        => PingState::Error,
                ctx.round() >= B::Round::from(MAX_ROUNDS)   => PingState::Done,
                ctx.round() == B::Round::from(PAUSE_AT_ROUND) => PingState::Paused,
                _                                           => PingState::Active,
            }
        },
    ],
};
```

### Correspondence

| Spec Element | Code Element |
|--------------|--------------|
| Handled by: `Active` | `ACTIVE_FNS` constant |
| Event: `PingPongMsg::Pong` | Pattern `PingPongMsg::Pong(_)` |
| Side effects: log, forward_ping | `actions [Self::log_pong_received, Self::forward_ping]` |
| Guard conditions | `guard(ctx, results) { ... }` block |
| Outcomes | Match arms returning `PingState::*` |

## Walkthrough: Acceptance Criterion → Test

### Spec (from ping.md)

```markdown
- [x] `PingPongMsg::Pong(n)` in `Active` with `round >= MAX_ROUNDS` transitions to `Done`
```

### Code (from tests.rs)

```rust
#[test]
fn test_pong_reaches_done_at_max_rounds() {
    let mut machine = make_ping_machine();
    let _ = machine.start();  // Enter Active (round 1)
    
    // Send Pong until we reach MAX_ROUNDS
    for _ in 1..MAX_ROUNDS {
        let ev = make_pong_event();
        let _ = machine.dispatch(ev);
    }
    
    // Final Pong should transition to Done
    let ev = make_pong_event();
    let outcome = machine.dispatch(ev);
    assert!(matches!(outcome, DispatchOutcome::Transition(PingState::Done)));
    
    // Verify is_terminal
    assert!(PingSpec::<TokioRuntime, PingBehavior>::is_terminal(&PingState::Done));
}
```

### Correspondence

| Spec Element | Code Element |
|--------------|--------------|
| "in Active" | Implicit — machine starts in Active |
| "with round >= MAX_ROUNDS" | Loop sends Pong until MAX_ROUNDS reached |
| "transitions to Done" | Assert on `DispatchOutcome::Transition(Done)` |

## Spec → Code Checklist

When implementing a new blox, verify each spec section has corresponding code:

- [ ] State Hierarchy → `#[derive(StateTopology)]` enum
- [ ] States table → `#[composite]` and `#[parent(...)]` annotations
- [ ] Events table → `transitions!` rules (one per event row)
- [ ] Context struct → `#[derive(BloxCtx)]` struct
- [ ] Context annotations → `#[self_id]`, `#[provides]`, `#[delegates]`
- [ ] Entry/Exit Actions → `on_entry: &[...]`, `on_exit: &[...]` slices
- [ ] Constants → `const` definitions at module level
- [ ] Acceptance Criteria → Test functions (one per criterion)

## Common Patterns

### Flat FSM (Counter style)
- States: All leaves
- Context: One `#[self_id]` + one `#[delegates]`
- Transitions: Direct `Transition(State)` or `stay`
- Tests: Simple dispatch assertions

### Composite with Siblings (Ping Operating style)
- States: One `#[composite]`, two+ `#[parent(...)]` children
- Context: Multiple `#[provides]` for ActorRefs
- Transitions: Composite handles "stray" events with `stay`
- Tests: Verify bubbling to composite

### Terminal State
- Spec: Mark as "terminal" in States table
- Code: `fn is_terminal(state: &State) -> bool { matches!(...) }`
- Tests: Assert `DispatchOutcome::Transition(s)` where `is_terminal(&s)`

### Error State
- Spec: Mark as "error" in States table
- Code: `fn is_error(state: &State) -> bool { matches!(...) }`
- Tests: Verify `is_error` takes precedence over `is_terminal`
